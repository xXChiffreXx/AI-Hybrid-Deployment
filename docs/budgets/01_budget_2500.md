# Architecture Option A: `$2,500` Machine Budget

## Goal

Minimize upfront cost while keeping a usable team SSH node with 10Gb networking and predictable knowledge-base throughput.

## Target Infrastructure Profile

- 1 shared research node
- 10Gb Ethernet
- 12-16 CPU cores
- 64-96 GB RAM
- 2 TB NVMe SSD
- GPU or unified-memory profile sufficient for quantized local models
- SSH access for small team (3-6 users)

This design is OS-agnostic because local inference standardizes on `ollama-server`.

## Logical Architecture

- `resolver-gateway` for per-resource orchestration, policy limits, and completion checks
- `ollama-server` for initial extraction and timed local fill attempts
- `cloud CLI adapter` for fallback fill calls when local thresholds fail
- `kb-writer` service writing canonical records to `SQLite + Parquet`
- optional `redis` and `qdrant` for cache/retrieval support outside core MVP flow

## Deployment Pattern

- Single-node deployment with one primary local inference worker pool
- Canonical write path enforced as `resolver-gateway -> kb-writer -> SQLite + Parquet`
- Per-resource termination on required-field completion with provenance

## MVP Runtime Flow (Aligned with Architecture Docs)

1. Ingest user-fed source plus required schema fields.
2. Run initial local extraction pass with `ollama-server`.
3. Upsert extracted values through `kb-writer` into canonical stores.
4. Enumerate missing required fields from SQL completion checks.
5. For each missing field, run local fill attempts with fixed `tau_local`.
6. If local timing/confidence thresholds fail, issue one cloud CLI fill call.
7. Re-enter cloud outputs via `resolver-gateway` and `kb-writer` with provenance.
8. Repeat completion checks until all required fields are complete, then terminate the resource job.

## Memory-Constrained Operating Point

Assumptions for Track A generation profile:

- Installed memory envelope: $M_{avail} = 64 \text{ to } 96$ GB
- Safety factor: $\eta = 0.85$
- Usable memory envelope: $\eta M_{avail} = 54.4 \text{ to } 81.6$ GB
- Planned runtime working set (`qwen2.5:14b`, quantized, medium context and concurrency): $M_{req} \approx 24 \text{ to } 40$ GB

Throughput decomposition:

- $\mu_{compute} \approx 125$ tokens/sec
- $\mu_{memory} \approx 90$ tokens/sec
- $f_{fit} = 1.00$

```math
\mu_{eff}=f_{fit}\cdot\min(\mu_{compute},\mu_{memory})=1.00\cdot 90=90
```

## Capacity Assumption

- Effective local capacity target: $\mu_{eff} = 90$ tokens/sec
- Per-source token load: $T_{in}+T_{out}=725{,}000$

Theoretical local-only source throughput:

```math
\text{sources/day} = \frac{\mu_{eff} \cdot 86400}{725000} \approx 10.73
```

## Cloud Cost Model (Instantiated)

Using the shared model in `00_cloud_cost_model.md`:

- Hard-task cloud fraction: $r_h = 0.20$
- Baseline peak demand: $\lambda_{peak}=44.75$ tokens/sec
- Stress peak demand: $\lambda_{peak}=80.56$ tokens/sec

Overflow fractions:

```math
r_{over,base} = \max\left(0,\frac{44.75-90}{44.75}\right)=0
```

```math
r_{over,stress} = \max\left(0,\frac{80.56-90}{80.56}\right)=0
```

Monthly cloud cost:

```math
C_{month}=S\left(r_hc_h+(1-r_h)r_{over}c_e\right)
```

For web-aware enrichment workflows, use the extended cost equation in `00_cloud_cost_model.md`:
$C_{month}^{web}$ with $h_{db}$, $p_{escalate}$, and $c_{fill}$.

with $c_e=0.33925$, $c_h=1.69625$.

Baseline (`S=40`):

```math
C_{base}=40\left(0.20\cdot1.69625+0.80\cdot0\cdot0.33925\right)=\$13.57/month
```

Stress (`S=72`):

```math
C_{stress}=72\left(0.20\cdot1.69625+0.80\cdot0\cdot0.33925\right)=\$24.43/month
```

Conservative web-aware planning assumptions for this tier:

- $h_{db}=0.60$
- $p_{escalate}=0.45$
- $c_{fill}=0.25$ USD/call

Web-aware cloud cost:

```math
C_{base}^{web}=40\left(0.20\cdot1.69625+0.80\cdot(1-0.60)\cdot0.45\cdot0.25\right)=\$15.01/month
```

```math
C_{stress}^{web}=72\left(0.20\cdot1.69625+0.80\cdot(1-0.60)\cdot0.45\cdot0.25\right)=\$27.02/month
```

Expected cloud calls/month under web-aware policy:

```math
N_{cloud\_calls,base}=40\left(0.20+0.80\cdot(1-0.60)\cdot0.45\right)=13.76
```

```math
N_{cloud\_calls,stress}=72\left(0.20+0.80\cdot(1-0.60)\cdot0.45\right)=24.77
```

Approximate annual cloud range from this model: `$163 - $293`.
Conservative web-aware annual cloud range: `$180 - $324`.

## Fit for Project

### Pros

- Low hardware cost with materially better memory headroom than the `$599` class.
- Fastest to acquire and deploy.
- Works for pilot and steady moderate throughput.

### Constraints

- Cloud costs remain higher than `$5,000` and `$7,500` tiers because of a larger hard-route fraction.
- Less SLO headroom than higher tiers during burst windows.
- Limited room for large-context or high-concurrency local serving compared with higher budgets.

## Recommended If

Choose this if your priority is low capital expense with stable throughput at `40/month` and you accept less burst headroom than higher tiers.
